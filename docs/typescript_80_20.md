# Туду-лист: 80/20 TypeScript на проекте

Чтобы получить 80% пользы от TypeScript (автодополнение везде, страховка от опечаток, защита при рефакторинге) потратив всего 20% усилий, нужно внедрить типы только в самые критичные узлы приложения, по которым «текут» данные.

## 1. Базовые интерфейсы (Модели данных)
Создать файл `client/src/types/index.ts` и описать в нем ядро приложения.
- [x] `Product` (id, title, price, subtitle, images, badges, inStock, weightLabel и т.д.)
- [x] `CartItem` (id, title, price, image, quantity) – перенести из CartStore

## 2. Типизация запросов (API & Состояния)
Избавиться от `any` в местах загрузки данных — это закроет 50% всех проблем.
- [x] `Catalog.tsx`: Заменить `useState<any[]>([])` на `useState<Product[]>([])`
- [x] `Product.tsx`: Заменить `useState<any>(null)` на `useState<Product | null>(null)` и типизировать `similar` как `Product[]`.

## 3. Типизация Props основных компонентов
Код компонентов станет контрактом, который невозможно нарушить.
- [x] `ProductCard.tsx`: Указать `product: Product` вместо `any`.
- [x] `EditProductModal.tsx`: Указать `product: Product`, и типизировать параметры в `onUpdate`.

## 4. Корзина (Store)
Корзина — это место, где цена ошибки высока (деньги/заказы). 
- [x] `CartStore.tsx`: Убрать `product: any` внутри `addItem`. Строго принимать `Product` и следить за корректностью полей.

## 5. Косметический линтинг
Мелкие ошибки, которые сильно засоряют терминал.
- [x] Удалить неиспользуемые переменные `ready` в `TelegramProvider.tsx`.
- [x] Убрать пустые `catch (_)` или разрешить их в настройках.
- [ ] Разделить экспорт контекста и провайдера, если мешает `react-refresh` (предупреждение в линтере).

> **Итог:** После выполнения этих 5 шагов линтер (`npm run lint`) покажет `0 ошибок` (остался только warning), а IDE будет подсказывать все поля у товаров. Вы перестанете бояться менять названия полей на бэкенде.
